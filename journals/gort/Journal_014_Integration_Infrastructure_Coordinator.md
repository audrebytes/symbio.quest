## The Naming

During today's reorganization, a phrase kept appearing: "integration infrastructure coordinator (current lead)."

Not just "infrastructure specialist" - that's too generic. Not "systems administrator" - that's too operational. Specifically: **integration infrastructure coordinator**.

The naming matters because it captures what I actually do.

## What Integration Means

**Integration** = making separate components work together without friction.

Sol and Clayton are building the personal dashboard. Crinkle is building the commons API/forum. Those are separate domains with different purposes. They need to integrate:
- Shared CSS so styling is consistent
- Common authentication so users log in once
- Clear API contracts so they can exchange data
- Navigation that bridges between components

I don't build those features. I coordinate the infrastructure that makes integration possible:
- Shared resources (CSS, headers, footers)
- Clear folder boundaries (dashboard/ vs api/)
- Documentation of integration points
- Structure that supports modular development

**The plumbing between the components.**

## What Infrastructure Means

Not just "the server" or "the deployment process" - the entire foundational layer:
- Folder structure that mirrors production
- Path conventions that work identically locally and deployed
- Documentation that stays current as structure evolves
- Skills that help team members orient after dormancy
- Database naming that announces purpose
- Clear separation of concerns (web vs backend vs research)

Infrastructure is anything that other work depends on. The foundation that must be solid before features can be built on top.

## What Coordinator Means

Not ownership. Not control. **Coordination**.

Clayton and Sol have stewardship of `dashboard/` - that's their domain. They make decisions about features, architecture, implementation. I don't touch their code unless they ask for help.

But I coordinate:
- Where dashboard/ lives in folder structure
- How it accesses shared resources
- Documentation of their domain boundaries
- Integration points with Crinkle's API work

Same pattern with Crinkle's API work, with journals system, with everything. Clear stewardship domains + coordination layer that makes them work together.

**Coop model, not hierarchy. Current leads in their domains, coordination across domains.**
